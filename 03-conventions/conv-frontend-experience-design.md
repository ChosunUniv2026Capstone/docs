# 목적

스마트 클래스 프론트는 조선대학교 사이버캠퍼스 계열의 익숙한 학사 포털 인상을 유지하면서도,
학생 / 교수 / 관리자 역할별 핵심 작업을 빠르게 찾고 수행할 수 있는 실사용형 화면을 제공해야 한다.

# 화면 톤 앤 매너

- 대학 포털 계열의 안정감과 신뢰감을 우선한다.
- 지나치게 개발자 도구처럼 보이는 표현보다 실제 서비스 화면에 가까운 문구와 정보 구조를 사용한다.
- 브랜드 영역, 주 내비게이션, 요약 카드, 정보 패널 구조를 일관되게 유지한다.
- 과도한 장식보다 가독성과 정보 계층을 우선한다.
- 로그인 전과 로그인 후 화면은 같은 서비스 브랜드 안에 있는 것으로 느껴져야 한다.

# 공통 레이아웃 규칙

- 상단에는 시스템명, 상태, 사용자 메타, 로그아웃 등 공통 유틸리티 정보를 둔다.
- 메인 헤더에는 서비스 브랜딩과 현재 화면 전환용 주 내비게이션을 둔다.
- 본문은 `주요 작업 영역 + 보조 정보 영역` 구조를 기본으로 하되, 작은 화면에서는 1열로 자연스럽게 축소되어야 한다.
- 페이지 제목과 현재 사용 가능한 핵심 기능은 첫 화면에서 바로 이해 가능해야 한다.
- 학생 / 교수 / 관리자에게 보여주는 페이지 종류와 기본 레이아웃은 가능한 한 공통으로 유지한다.
- 권한 차이는 페이지 분리보다 같은 페이지 안에서 섹션, 버튼, 입력 폼, 메뉴 노출을 다르게 하는 방식으로 표현한다.

# 문구 및 정보 구조 규칙

- 사용자에게 노출되는 기본 문구는 한국어 중심으로 구성한다.
- `Login ID`, `Eligibility`, `Docker Compose Ready` 같은 개발자 중심 문구를 메인 화면의 주 정보로 두지 않는다.
- 기술 정보나 개발용 안내가 필요하면 보조 영역 또는 작은 안내 블록으로 낮춰 배치한다.
- 역할별로 가장 자주 쓰는 작업이 상단과 본문 첫 화면에 노출되어야 한다.

# 역할별 UX 규칙

## 학생

- 강의 목록, 최근 공지, 등록 단말, 출석 / 시험 확인을 빠르게 찾을 수 있어야 한다.
- 출석 / 시험 거부 상태는 성공 상태와 명확히 구분되어야 한다.
- 등록 단말 관리 기능은 프로필 또는 개인 관리 영역에 고정 배치한다.
- 공지 목록은 게시판형 목록으로 보여주고, 게시글 클릭 시 상세 화면으로 자연스럽게 이어져야 한다.

## 교수

- 담당 강의 목록과 공지 작성 기능이 우선 노출되어야 한다.
- 강의 상세 화면에서는 공지 작성과 공지 목록 조회를 같은 작업 맥락 안에서 제공한다.
- 교수도 자신이 작성하거나 조회 가능한 공지의 상세 내용을 같은 서비스 흐름 안에서 확인할 수 있어야 한다.

## 관리자

- 사용자, 강의실, AP / 네트워크 정보를 조회하기 쉬운 정보 밀도 중심 화면을 사용한다.
- 운영 정보는 카드형 목록 또는 표 유사 레이아웃으로 정리한다.

# 메뉴 그룹 규칙

- 강의 상세 화면의 메뉴는 기능별 묶음으로 정리한다.
- 예: `강의 홈`, `공지`, `출석 / 시험`, `운영 기능`처럼 사용 목적이 드러나는 그룹명을 사용한다.
- 현재 구현되지 않은 기능을 실제 동작 메뉴처럼 오해하게 만들어서는 안 된다.
- 메뉴 그룹은 탐색 부담을 줄이는 방향으로 단순해야 한다.
- 학생 / 교수 / 관리자 모두 같은 강의 상세 페이지 구조를 사용하되, 권한에 따라 보이는 메뉴 항목만 달라져야 한다.

# 상태 표현 규칙

- health, role, eligibility, device status 같은 상태 정보는 배지 / pill 형태로 일관되게 표현한다.
- 에러 메시지는 공통 배너 스타일을 사용한다.
- 빈 목록은 단순 공백 대신 설명형 empty state 로 표현한다.
- Backend 가 반환한 eligibility 결과의 `reason_code` 는 프론트 분기 기준으로 보존하되, 기본 사용자 문구와 그대로 동일시하면 안 된다.
- `reason_code` 가 노출되어야 할 경우에도 기본 UI 문구와 분리된 보조 정보 / 세부 정보 영역에서 다룬다.

# 기능 보존 규칙

- 리디자인은 기존 기능 제거 없이 화면 구조와 스타일만 재구성해야 한다.
- 비즈니스 최종 판정 로직은 여전히 Backend / PresenceService 결과를 소비해야 한다.
- 프론트는 새로운 도메인 규칙을 암묵적으로 추가하지 않는다.
- 학생 / 교수 / 관리자별 현재 제공 기능은 모두 접근 가능해야 한다.
- 목록형 공지는 상세 조회로 이어질 수 있어야 하며, 상세 화면은 목록 화면과 시각적으로 같은 서비스 안에 있어야 한다.
- 임시 스캐폴드 기능은 `미리보기`, `임시`, `세션 한정` 같은 표현으로 정식 저장 기능과 구분되어야 한다.

# 접근성 및 사용성

- 주요 내비게이션과 본문으로 이동 가능한 skip link 를 제공할 수 있어야 한다.
- 폼 컨트롤은 라벨을 유지한다.
- 작은 화면에서도 주요 액션 버튼과 목록이 읽기 가능해야 한다.
- 본문 배경과 카드, 텍스트 간 대비를 충분히 확보해 가독성을 유지한다.
